  1. 프로젝트 개요
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  이 시스템은 카메라로 방 안을 촬영하고 Vision Language Model(VLM)로 영상을 분석한 뒤,
  ISO 7730 국제표준의 PMV(Predicted Mean Vote, 예측 평균 온열감) 모델을 통해
  재실자가 느끼는 온열 쾌적성을 계산하여, 공조기(에어컨)를 자동으로 제어하는 시스템이다.

  최종 목표 하드웨어: NVIDIA Jetson Orin Nano Super (8GB)
  최종 목표 모델   : Qwen2.5-VL 3B (INT4 양자화 + TensorRT 가속)
  현재 개발 환경   : Windows 노트북 (CPU only), Qwen2-VL-2B-Instruct 모델


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  2. 에어컨 시뮬레이터를 사용한 이유
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  [이유 1] 실제 에어컨과의 통신 인터페이스가 없다
  ─────────────────────────────────────────────────
  실제 에어컨을 소프트웨어로 제어하려면 제조사가 제공하는 API, 시리얼 통신(RS-485),
  또는 스마트홈 프로토콜(Matter, Zigbee, IR 블래스터 등)이 필요하다.
  현재 개발 단계에서는 이런 하드웨어 인터페이스가 준비되어 있지 않기 때문에,
  소프트웨어만으로 에어컨의 동작을 수학적으로 흉내내는 시뮬레이터를 만들었다.

  [이유 2] 실내 온습도 센서도 없다
  ─────────────────────────────────────────────────
  PMV 계산에는 실내 공기 온도(ta)와 실내 상대 습도(rh)가 필수 입력값이다.
  실제 IoT 센서(DHT22, SHT31 등)가 없는 상태에서도 시스템 전체를 테스트할 수 있도록,
  시뮬레이터가 '현재 실내 온도'와 '현재 실내 습도'를 계산해서 제공한다.

  [이유 3] 제어 명령의 효과를 시각적으로 확인하기 위해
  ─────────────────────────────────────────────────────
  "냉방 강화 명령을 내렸을 때 실내 온도가 실제로 내려가는가?"를 확인하려면
  응답하는 물리계가 필요하다. 시뮬레이터는 에어컨 설정값에 따라 실내 온도와 습도를
  매 프레임마다 업데이트함으로써, 제어 명령의 효과를 실시간으로 수치로 확인할 수 있게 한다.

  [이유 4] 1차 구현 목표: 알고리즘 검증이 우선
  ─────────────────────────────────────────────────
  VLM 분석 → PMV 계산 → 제어 명령 생성 이라는 핵심 파이프라인이 올바르게 동작하는지
  먼저 소프트웨어 레벨에서 검증한 뒤, 추후 Jetson + 실제 센서/에어컨으로 포팅할 계획이다.
  시뮬레이터는 이 검증 단계를 위한 '가상의 물리 환경' 역할을 한다.

  [결론]
  시뮬레이터가 하는 일은 크게 두 가지다.
    (A) 에어컨 역할: 설정 온도/풍량에 따라 실내 온도를 낮추고 제습한다.
    (B) 실내 센서 역할: 현재 실내 온도와 습도 값을 계산해서 제공한다.
  실제 시스템에서는 (A)는 에어컨 API로, (B)는 IoT 센서로 대체된다.

  │            - EMPTY → 자동 닫기                             │
  │            - 자연환기 조건 충족 → 자동 열기                 │
  │                 │                                          │
  │                 ▼ power, target_temp, fan_speed, window    │
  │         [hvac_simulator.py]                                │

  │                 │                                          │

  4-1. weather_service.py  [외부 날씨 데이터 수집]
  ────────────────────────────────────────────────────────────────

  역할:
    OpenWeatherMap API에 HTTP GET 요청을 보내 현재 위치의 실외 기상 데이터를 수신한다.
    city 이름 방식(q=Saha-gu)은 행정구(district) 단위를 인식하지 못해 404 오류가 발생하므로,
    위도/경도 좌표 기반 쿼리(lat=, lon=)를 사용한다.
    API 호출이 실패할 경우, 마지막으로 성공한 값 또는 기본값(20°C, 50%)을 반환한다.

  API 엔드포인트:
    https://api.openweathermap.org/data/2.5/weather
      ?lat={위도}&lon={경도}&appid={API키}&units=metric

  생성 변수 및 의미:
  ┌──────────────┬────────────┬─────────────────────────────────────────────┐
  │  변수명       │  단위      │  설명                                       │
  ├──────────────┼────────────┼─────────────────────────────────────────────┤
  │  out_temp    │  °C        │  현재 실외 기온                             │
  │  out_humid   │  %         │  현재 실외 상대 습도                        │
  │  out_weather │  문자열    │  날씨 상태 (예: "clear sky", "light rain") │
  │  out_wind    │  m/s       │  실외 풍속 (향후 외풍 계산 활용 가능)       │
  └──────────────┴────────────┴─────────────────────────────────────────────┘

  기본값 (API 실패 시):
    out_temp = 20.0°C, out_humid = 50.0%, out_weather = "unknown", out_wind = 0.0 m/s

  main.py에서 호출하는 방법:
    out_temp, out_humid, out_weather, out_wind = weather.fetch_current_weather()

  이 값이 사용되는 곳:
    - hvac_simulator.simulate_step(out_temp, out_humid, ...) → 실내 온도 평형 계산에 사용
    - CSV 로그에 저장


  ────────────────────────────────────────────────────────────────
  4-2. vlm_processor.py  [VLM 비전 분석]
  ────────────────────────────────────────────────────────────────

  역할:
    카메라에서 캡처된 OpenCV 프레임(BGR 이미지)을 입력받아
    Qwen2-VL-2B-Instruct 비전 언어 모델(Vision Language Model)로 분석하고,
    PMV 계산에 필요한 착의량(Clo)과 대사율(Met), 인원수를 반환한다.

  처리 과정:
    ① 프레임을 160×160으로 축소 (CPU 부하 절감, Windows 노트북 최적화)
    ② BGR → RGB 변환 후 PIL 이미지로 변환
    ③ 모델에 JSON 형식 응답을 요청하는 프롬프트와 함께 이미지 입력
       프롬프트 내용:
         "sleeves"     : 소매 길이 ('long' 또는 'short')
         "outerwear"   : 아우터 착용 여부 ('yes' 또는 'no')
         "activity"    : 활동 유형 ('lying', 'sitting', 'standing', 'walking', 'cooking', 'exercising')
         "bags"        : 가방 소지 여부 ('yes' 또는 'no')  ← 퇴근 맥락 감지 신호
         "heat_source" : 조리기구 등 열원 존재 ('yes' 또는 'no') ← tr 보정 및 환기 신호
         "people"   : 화면 내 인원 수 (정수)
    ④ 모델 응답 텍스트에서 정규표현식으로 JSON 파싱
    ⑤ 파싱 결과를 PMV 입력값으로 변환 (아래 매핑 참조)

  VLM 출력 → PMV 파라미터 매핑:

    [착의량 (Clo) 매핑]  - ISO 7730 기준
    ┌─────────────────────────────────┬──────────┐
    │  VLM 분류                        │  Clo 값  │
    ├─────────────────────────────────┼──────────┤
    │  sleeves=short                  │  0.5     │
    │  sleeves=long                   │  1.0     │
    │  outerwear=yes 시 추가           │  +0.3    │
    └─────────────────────────────────┴──────────┘
    예시: 긴팔 + 아우터 → Clo = 1.0 + 0.3 = 1.3

    [대사율 (Met) 매핑]  - ISO 7730 기준
    ┌─────────────────────────────────┬──────────┐
    │  activity 값                     │  Met 값  │
    ├─────────────────────────────────┼──────────┤
    │  lying (누워있음)                │  0.8     │  ← 신규 추가
    │  sitting (착석)                  │  1.0     │
    │  standing (기립)                 │  1.2     │
    │  walking (보행)                  │  1.5     │
    │  cooking (조리)                  │  2.0     │
    │  exercising (운동)               │  3.0     │
    │  인식 불가 시 기본값              │  1.2     │
    └─────────────────────────────────┴──────────┘

  반환값:
    {"clo": float, "met": float, "count": int,
     "bags": str, "heat_source": str, "outerwear": str, "activity": str}
    분석 실패 시 None 반환 (main.py에서 경고 출력 후 건너뜀)

  생성 변수:
  ┌─────────────┬────────┬────────────────────────────────────────────────────┐
  │  변수명      │ 단위   │  설명                                              │
  ├─────────────┼────────┼────────────────────────────────────────────────────┤
  │  clo        │  clo   │  착의 단열값. PMV 계산의 착의량 입력               │

  │  count      │  명    │  화면 내 감지된 인원 수. 제어 로직 및 시뮬레이터 입력│
  │  bags       │  문자열│  가방 소지 여부('yes'/'no'). 퇴근 맥락 점수 +25   │

  4-3. hvac_simulator.py  [에어컨 + 실내 환경 시뮬레이터]
  ────────────────────────────────────────────────────────────────

  역할:
    (A) 가상 에어컨: 설정된 목표 온도와 풍량에 따라 실내 온도를 변화시킨다.
    (B) 가상 실내 센서: 매 스텝마다 실내 온도/습도를 계산해서 제공한다.
    실제 배포 시 (A)는 에어컨 API로, (B)는 DHT22/SHT31 같은 IoT 센서로 교체된다.

  내부 상태 변수:
  ┌──────────────┬────────┬───────────────────────────────────────────────────┐
  │  변수명       │ 단위   │  설명                                             │
  ├──────────────┼────────┼───────────────────────────────────────────────────┤
  │  indoor_temp │  °C    │  현재 실내 공기 온도 (초기값: 27.0°C)            │
  │  indoor_humid│  %     │  현재 실내 상대 습도 (초기값: 60.0%)             │
  │  room_size   │  m²    │  방 면적 (기본값: 20.0m²)                        │
  │  window_open │  bool  │  창문 개폐 여부 (True=열림, False=닫힘)           │
  │  is_on       │  bool  │  에어컨 전원 상태                                │
  │  target_temp │  °C    │  에어컨 설정 온도 (기본값: 24.0°C)              │
  │  fan_speed   │  1~3   │  풍량 (1=약, 2=중, 3=강)                        │
  └──────────────┴────────┴───────────────────────────────────────────────────┘

  simulate_step() 내부 계산 로직:

    매 프레임(약 33ms)마다 호출되며, 다음 순서로 실내 온도/습도를 업데이트한다.

    [Step 1] 방 크기 보정계수 계산
      size_factor = 20.0 / max(room_size, 5.0)
      → 방이 기준(20m²)보다 클수록 온도 변화가 느려진다.
      예시: 40m² 방이면 size_factor = 0.5 (냉방 효율 절반)

    [Step 2] 에어컨 냉방 적용 (is_on=True이고 indoor_temp > target_temp일 때만)
      indoor_temp  -= 0.02 × fan_speed × size_factor
      indoor_humid -= 0.05  (제습 효과)
      예시: fan_speed=3, size_factor=1.0이면 → 매 스텝 0.06°C 하강

    [Step 3] 외기와의 온도 평형
      equil_rate = 0.05 (창문 열림) 또는 0.005 (창문 닫힘)
      indoor_temp += (outdoor_temp - indoor_temp) × equil_rate
      → 창문이 열리면 외기 영향이 10배 강해진다.
      → 에어컨이 켜져 있어도 이 평형은 항상 발생한다. (현실 반영)

    [Step 4] 창문 열릴 때 외부 습도 영향
      indoor_humid += (outdoor_humid - indoor_humid) × 0.05
      → 창문이 열리면 외부 습도 방향으로 서서히 수렴한다.

    [Step 5] 재실 인원 체열 반영
      indoor_temp += people_count × 0.03 × size_factor
      → 1인당 약 70W 발열 기준으로 근사. 방이 작을수록(size_factor 큼) 영향이 크다.

    [Step 6] 물리적 범위 클램핑
      indoor_temp  = clamp(10.0 ~ 45.0)
      indoor_humid = clamp(10.0 ~ 95.0)

  제어 명령 인터페이스:
    set_control(power=True/False, target=온도, fan=1~3)
    → main.py의 제어 로직 결과를 받아 에어컨 설정값을 갱신한다.
    → 다음 simulate_step()부터 새 설정이 반영된다.

  공간 설정 인터페이스:
    set_room(size_m2=방크기, window_open=True/False)
    → 공간 특성을 일괄 업데이트한다. 프로그램 시작 시 main.py에서 호출.

  4-4. thermal_engine.py  [ISO 7730 PMV 열쾌적성 계산]
  ────────────────────────────────────────────────────────────────

  역할:
    ISO 7730:2005 및 ASHRAE 55 표준에 기반한 PMV(Predicted Mean Vote) 계산.
    PMV는 어떤 환경에서 사람이 느끼는 온열감을 -3(매우 춥다)~+3(매우 덥다)으로 표현한 지수다.

  입력 파라미터:
  ┌──────┬──────────────────────┬────────┬─────────────────────────────────────┐
  │ 변수 │  의미                 │ 단위   │  출처                               │
  ├──────┼──────────────────────┼────────┼─────────────────────────────────────┤
  │  ta  │  실내 공기 온도       │  °C    │  hvac_simulator.indoor_temp         │
  │  tr  │  평균 복사 온도       │  °C    │  현재는 ta와 동일하게 근사          │
  │  rh  │  실내 상대 습도       │  %     │  hvac_simulator.indoor_humid        │
  │  vel │  실내 기류 속도       │  m/s   │  fan_speed → FAN_VELOCITY 딕셔너리  │
  │  met │  대사율               │  met   │  vlm_processor 분석 결과            │
  │  clo │  착의량               │  clo   │  vlm_processor 분석 결과            │
  └──────┴──────────────────────┴────────┴─────────────────────────────────────┘

  풍량 → 기류 속도 변환표 (main.py의 FAN_VELOCITY):
    fan_speed = 1 (약) → vel = 0.1 m/s
    fan_speed = 2 (중) → vel = 0.3 m/s
    fan_speed = 3 (강) → vel = 0.5 m/s

  내부 계산 단계:

    [Step 1] 입력 범위 강제 클램핑 (수치 발산 방지)
      ta:  10.0 ~ 40.0°C
      tr:  10.0 ~ 50.0°C
      rh:  0.0  ~ 100.0%
      vel: 0.0  ~ 5.0 m/s
      met: 0.8  ~ 4.0 met
      clo: 0.0  ~ 2.0 clo

    [Step 2] 수증기 분압(pa) 계산
      pa = (rh/100) × 10 × exp(16.6536 - 4030.183 / (ta + 235))  [kPa]

    [Step 3] 기초 변환
      icl = 0.155 × clo          (착의 열저항, m²·K/W)
      m   = met × 58.15          (대사율, W/m²)
      fcl = 착의 표면적 계수 (icl에 따른 구분)

    [Step 4] 착의 표면 온도(tcl) 반복 수렴
      열 복사와 대류의 균형으로부터 tcl을 구하는 방정식은 비선형이라
      해석적 풀이가 불가능하다. 따라서 반복법(최대 150회)으로 수치 해를 구한다.
      수렴 조건: |tcl_new - tcl| < 0.0001°C

    [Step 5] 6개 열손실 항목 계산 (ISO 7730 Table C.1 기준)
      hl_skin   : 피부 확산에 의한 열손실
      hl_sweat  : 발한에 의한 열손실 (mw > 58.15W/m² 일 때만)
      hl_res_lat: 호흡 잠열 손실
      hl_res_dry: 호흡 현열 손실
      hl_rad    : 복사 열손실
      hl_conv   : 대류 열손실

    [Step 6] PMV 계산
      ts  = 0.303 × exp(-0.036 × m) + 0.028  (감각 계수)
      L   = mw - (hl_skin + hl_sweat + hl_res_lat + hl_res_dry + hl_rad + hl_conv)
      PMV = ts × L

    [Step 7] 최종 클램핑 및 NaN/Inf 처리
      PMV = clamp(-3.0 ~ 3.0)
      NaN 또는 Inf 발생 시 0.0 (쾌적, Neutral) 반환

  출력값:
    float: -3.0 ~ +3.0 (소수점 둘째 자리)

  PMV → 쾌적감 분류:
  ┌────────────────────┬────────────────────────────────┐
  │  PMV 범위           │  상태                          │
  ├────────────────────┼────────────────────────────────┤
  │  +2.5 초과          │  매우 더움 (Hot)               │
  │  +1.5 ~ +2.5        │  더움 (Warm)                   │
  │  +0.5 ~ +1.5        │  조금 더움 (Slightly Warm)     │
  │  -0.5 ~ +0.5        │  쾌적 (Neutral)  ← 목표 범위  │
  │  -1.5 ~ -0.5        │  조금 추움 (Slightly Cool)     │
  │  -2.5 ~ -1.5        │  추움 (Cool)                   │
  │  -2.5 미만          │  매우 추움 (Cold)               │
  └────────────────────┴────────────────────────────────┘

    ② HUD(현재 실내 온도/습도/에어컨 상태/창문 상태)를 영상에 오버레이

  제어 로직 (decide_control 함수):
    PMV > +0.5 (더움):
      → 목표 온도 = 22°C - (인원-1) × 0.5°C  (최대 2°C 추가 하향)
      → 풍량 = 인원 3명 이상이면 3, 아니면 2
      → 에어컨 ON
    PMV < -0.5 (추움):
      → 목표 온도 = 26°C, 풍량 = 1 (절전/난방 모드)
      → 에어컨 ON
    -0.5 ≤ PMV ≤ +0.5 (쾌적):
      → 설정 변경 없음 (현재 상태 유지)

    'w' 키  → 창문 수동 개폐 토글 (닫힘 ↔ 열림)
               열원 감지/EMPTY 등 자동 창문 제어가 있지만 수동 키로도 덮어쓸 수 있다.
    'q' 키  → 프로그램 종료 (에너지 리포트 자동 출력 후 종료)

  [화면 HUD 2줄]
    1줄: Temp:27.0C  Humid:60.0%  AC:ON(22C/Fan2)  Win:O
    2줄: State:STEADY  Score:0  Save:15.3%  Comfort:82.0%  Power:1200W

    State   : 현재 시스템 상태 (EMPTY/ARRIVAL/STEADY/PRE_DEPARTURE)
    Score   : 퇴근 맥락 점수 (0~100, 70 이상이면 PRE_DEPARTURE 전환)
    Save    : 베이스라인 대비 에너지 절약률 (%)
    Comfort : PMV ±0.5 이내 유지 비율 (%)
    Power   : 현재 소비전력 (W)

  [분석 주기]
    's' 키를 누를 때마다 한 번씩 분석이 실행된다.
    CPU에서 VLM 추론은 약 10~60초 소요된다. (노트북 성능에 따라 다름)
    VLM 분석 중에도 화면은 계속 갱신되고, 시뮬레이터 + 에너지 모니터는 계속 실행된다.

  [결과 확인]
    분석 실행 시마다 hvac_system_performance.csv 파일에 한 행씩 추가된다.
    Excel 또는 pandas로 열어서 PMV 추이, 제어 이력, 에너지 절약 데이터를 확인할 수 있다.

    CSV 주요 신규 컬럼:
      system_state    : 각 분석 시점의 시스템 상태
      departure_score : 퇴근 맥락 점수
      power_w         : 해당 시점 소비전력 (W)
      energy_kwh      : 누적 실제 소비 전력량 (kWh)
      baseline_kwh    : 누적 베이스라인 전력량 (kWh)
      savings_pct     : 누적 절약률 (%)
      comfort_rate    : PMV 쾌적 유지율 (%)

  8. 상태 머신 상세 설명 (State Machine)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  파일: state_machine.py
  클래스: StateManager

  [왜 상태 머신이 필요한가?]
    기존 시스템은 매 프레임을 독립적으로 판단했다. 즉, 사람이 방금 도착했는지,
    1시간째 있었는지, 곧 나갈 것인지 전혀 알 수 없었다. 이는 단순한 온도계와 다름없다.
    상태 머신은 '지금 어떤 상황인가'를 시간 흐름 속에서 추적하여 맥락에 맞는 제어를 한다.

  [4가지 상태]
  ┌──────────────────┬────────────────────────────────────────────────────────┐
  │  상태             │  의미 및 제어 동작                                     │
  ├──────────────────┼────────────────────────────────────────────────────────┤
  │  EMPTY           │  인원 없음. 공조기 OFF. 창문 닫기. 완전 절전.          │
  │  ARRIVAL         │  인원 막 도착 (첫 10분). 빠른 냉·난방. 풍량 최대.      │
  │                  │  여름(외기>25°C): 목표 20°C, 풍량 3                    │
  │                  │  겨울(외기<10°C): 목표 26°C, 풍량 3                    │
  │                  │  봄/가을: 목표 22°C, 풍량 2                            │
  │  STEADY          │  안정 운전. ISO 7730 PMV 기반 미세 제어.               │
  │                  │  PMV>0.5: 냉방 강화 (인원수 반영 목표온도 조정)         │
  │                  │  PMV<-0.5: 난방/절전                                   │
  │                  │  -0.5≤PMV≤0.5: 현재 유지 (에너지 절약)                │
  │  PRE_DEPARTURE   │  퇴근 맥락 감지. 목표 25°C, 풍량 1 (잔열·잔냉 활용).  │
  │                  │  인원이 0이 되면 즉시 EMPTY 전환 → OFF.                │
  └──────────────────┴────────────────────────────────────────────────────────┘

  [상태 전이 다이어그램]
    ┌─────────┐  인원 감지         ┌─────────┐
    │  EMPTY  │ ────────────────► │ ARRIVAL │
    │         │ ◄────────────────  │         │
    └─────────┘  인원 0, 30초 경과 └────┬────┘
          ▲                             │ 10분 경과
          │                             ▼
          │               인원 0    ┌─────────┐
          └─────────────────────── │  STEADY │
          │    (30초 후)            └────┬────┘
          │                             │ 퇴근 점수 ≥ 70
          │                             ▼
          │                      ┌──────────────┐
          └──── 인원 0 (즉시) ── │PRE_DEPARTURE │
                                  └──────────────┘
                                   ↕ 점수 < 40: STEADY 복귀

  [퇴근 맥락 점수 계산 (0~100)]
  ┌────────────────────────────────────────────────┬────────┐
  │  조건                                           │  점수  │
  ├────────────────────────────────────────────────┼────────┤
  │  인원 수가 이전 분석보다 감소                   │  +30   │
  │  VLM outerwear = 'yes' (아우터 착용)           │  +25   │
  │  VLM bags = 'yes' (가방 소지)                  │  +25   │
  │  VLM activity = 'standing' (서있음)            │  +10   │
  │  현재 시각 = 퇴근 시각 ±1시간 이내             │  +10   │
  └────────────────────────────────────────────────┴────────┘
  70점 이상 → PRE_DEPARTURE 전환
  40점 미만 → STEADY 복귀 (오탐 방지 히스테리시스)

  [main.py에서 사용 방법]
    sm = StateManager(work_start_hour=9, work_end_hour=18)

    # VLM 분석 후 상태 갱신
    current_state = sm.update(
        people_count=vlm_data['count'],
        outerwear=vlm_data['outerwear'],
        activity=vlm_data['activity'],
        bags=vlm_data['bags'],
    )
    # 퇴근 맥락 점수 확인
    print(sm.departure_score)

  9. 에너지 모니터 상세 설명 (Energy Monitor)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  파일: energy_monitor.py
  클래스: EnergyMonitor

  [왜 에너지 모니터가 필요한가?]
    이 프로젝트의 핵심 주장 중 하나는 "에너지를 절약한다"는 것이다.
    논문과 보고서에서 이를 증명하려면 구체적인 수치(kWh, 절약률 %)가 필요하다.
    에너지 모니터는 본 시스템과 전통 방식(항상 Fan2 가동)을 동시에 시뮬레이션하여
    절약량을 자동으로 계산하고 CSV에 기록한다.

  [소비전력 모델 (가정치)]
  ┌─────────────────────────────┬──────────┐
  │  운전 모드                   │  소비전력 │
  ├─────────────────────────────┼──────────┤
  │  공조기 OFF                  │    0 W   │
  │  Fan 1 (약)                  │  800 W   │
  │  Fan 2 (중) - 베이스라인     │ 1200 W   │
  │  Fan 3 (강)                  │ 1600 W   │
  └─────────────────────────────┴──────────┘
  실측 후 energy_monitor.py의 POWER_W 딕셔너리를 수정하면 더 정확한 값 적용 가능.

  [베이스라인 정의]
    전통 방식: 재실자가 있는 동안 Fan 2(1200W)를 상시 가동.
    (VLM/PMV 없이 단순 온도 설정만 유지하는 구형 에어컨 방식)

  [논문 핵심 지표]
    절약률(%)    = (베이스라인 kWh - 실제 kWh) / 베이스라인 kWh × 100
    쾌적 유지율  = PMV ∈ (-0.5, 0.5) 인 분석 횟수 / 전체 분석 횟수 × 100

    이 두 지표를 함께 제시하면 "에너지는 절약하면서도 쾌적성은 유지한다"는
    시스템의 핵심 가치를 정량적으로 증명할 수 있다.

  [매 프레임 tick() 동작]
    em.tick(is_on, fan_speed, people_count, pmv_val)
    → 실제 소비 전력량 누적 (Wh)
    → 베이스라인 전력량 누적 (재실자 있을 때만)
    → PMV 쾌적 범위 샘플 카운팅

  [제공하는 메서드]
    em.get_energy_kwh()     → 실제 누적 소비 전력량 (kWh)
    em.get_baseline_kwh()   → 베이스라인 누적 전력량 (kWh)
    em.get_savings_pct()    → 에너지 절약률 (%)
    em.get_comfort_rate()   → PMV 쾌적 유지율 (%)
    em.get_current_power_w()→ 현재 소비전력 (W)
    em.print_summary()      → 종료 시 전체 리포트 출력
