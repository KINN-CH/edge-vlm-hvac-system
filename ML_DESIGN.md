## 1. 배경 및 목적

### 현재 시스템의 한계

현재 시스템은 VLM(Qwen2-VL-2B)과 YOLOv8n을 활용하지만, 이는 **사전학습된 외부 모델을 가져다 쓰는 것**이다.
제어 로직의 핵심(상태 전이, 퇴근 감지, 선호도 반영)은 여전히 **고정된 규칙(Rule-based)**으로 동작한다.

```
현재:
  퇴근 감지   → 고정 가중치 (+30/+25/+10/+10) 합산
  선호도 반영 → 버튼 클릭 시 ±0.5 고정 단계 누적
  재실 대응   → 감지된 후 반응 (사후 대응)
```

### 목표

**우리가 직접 설계·학습한 경량 ML 모델 3개**를 추가하여 규칙 기반 한계를 극복한다.

```
추가 후:
  퇴근 감지   → 우리가 학습한 분류기가 판단
  선호도 반영 → 개인 사용 패턴을 학습한 MLP가 예측
  재실 대응   → LSTM이 도착 전 예측 → 선제 냉난방
```

## 3. 모듈 ① — PMV 선호도 온라인 학습 MLP

### 3.1 목적

같은 PMV 값이어도 사람마다 체감이 다르다.
버튼 클릭 이력을 학습해 **"이 환경·이 사람은 PMV -0.2일 때 춥다고 느낀다"** 를 자동으로 파악한다.

### 3.2 현재 방식 vs ML 방식

```
현재:
  추워요 클릭 → pref_value += 0.5  (무조건 동일 단계)
  더워요 클릭 → pref_value -= 0.5

ML 적용 후:
  현재 PMV + 온도 + 습도 + 시간대 → MLP → 예상 선호도 오프셋 예측
  클릭 발생 시 → 실제값으로 모델 즉시 업데이트 (온라인 학습)
```

### 3.3 모델 구조

```
입력 (6개 features):
  pmv_now       : 현재 PMV 값 (-3 ~ +3)
  indoor_temp   : 실내 온도 (°C)
  indoor_humid  : 실내 습도 (%)
  hour          : 현재 시각 (0~23, sin/cos 인코딩)
  season        : 계절 (0~3, one-hot)
  state         : 현재 상태 (STEADY=0, ARRIVAL=1, ...)

모델:
  Linear(6 → 32) → ReLU → Linear(32 → 16) → ReLU → Linear(16 → 1)
  출력: 선호도 오프셋 (-2.0 ~ +2.0)
  파라미터 수: ~700개 (< 10KB)

손실 함수: MSE(예측 오프셋, 실제 클릭 방향)
옵티마이저: SGD (lr=0.01)
```

### 3.4 학습 방법

**온라인 학습 (Online Learning)** — 버튼 클릭 시마다 즉시 1 step 업데이트

```
트리거: 추워요/더워요 버튼 클릭
label 생성:
  추워요 클릭 → target = +0.5 (더 따뜻하게 원함)
  더워요 클릭 → target = -0.5 (더 시원하게 원함)
학습: forward → loss 계산 → backward → weight 업데이트
소요 시간: < 1ms (CPU에서도 무시 가능)
저장: models/pmv_preference.pt (클릭마다 덮어씀)
```

### 3.5 추론 방법

```
추론 시점: 매 제어 루프 (30초마다)
입력: 현재 센서값 + 상태
출력: predicted_offset (float)
적용: adjusted_pmv = pmv_now - predicted_offset
```

### 3.6 콜드 스타트

모델 파일 없을 때 → 기존 룰베이스 ±0.5 방식으로 fallback
클릭 10회 이상 누적 후 → ML 모델로 자동 전환

### 3.7 학습 데이터 확보

- 실제 사용자 클릭부터 즉시 시작 가능
- 별도 데이터 수집 기간 불필요

## 4. 모듈 ② — 재실 패턴 예측 LSTM

### 4.1 목적

현재는 사람이 도착한 **후**에 ARRIVAL 상태로 전환해 냉난방을 시작한다.
LSTM이 **도착 30분 전**에 예측해 미리 쾌적한 환경을 만든다.

```
현재: 도착 감지 → ARRIVAL → 10분 집중 냉난방 → STEADY
개선: 30분 전 예측 → 미리 온도 조정 → 도착 즉시 STEADY 수준
```

### 4.2 모델 구조

```
입력 시퀀스 (과거 48시간, 30분 단위 → 96 timesteps):
  각 timestep: [인원수, 시간(sin/cos), 요일(sin/cos)]  → 5 features

LSTM:
  input_size  = 5
  hidden_size = 32
  num_layers  = 1
  output      = Linear(32 → 1) → Sigmoid
  출력: 30분 뒤 재실 확률 (0.0 ~ 1.0)
  파라미터 수: ~5,000개 (< 100KB)
```

### 4.3 요일 처리 방식

요일을 별도 모델로 나누지 않고 **sin/cos 인코딩으로 feature에 포함**

```python
day_sin = sin(2π × weekday / 7)
day_cos = cos(2π × weekday / 7)
```

모델 1개가 요일 패턴까지 학습 → 데이터 효율 7배

### 4.4 학습 데이터 확보 전략 — Sim-to-Real

**실제 데이터가 없어도 사전 학습 가능**

```
1단계 (지금): 환경 프로파일 기반 시나리오 데이터 생성
  사무실 시나리오:
    월~금 09:00~18:00  → 인원 2~5 (가우시안 노이즈)
    월~금 12:00~13:00  → 인원 0~1 (점심)
    토~일              → 인원 0   (가끔 1)
  수천 일치 데이터 → 수 초 생성 가능

2단계 (운영 2~4주 후): 실제 수집 데이터로 재학습 (fine-tuning)
  새벽 3시 스케줄러가 자동 실행
  성능 향상 시에만 모델 교체 (롤백 보호)
```

### 4.5 학습 주기

```
초기: 시나리오 데이터로 사전 학습 → 배포
운영 중: 매일 새벽 3시 자동 재학습
  - 최근 30일 실데이터 사용
  - validation loss 비교 → 기존 모델보다 나을 때만 교체
```

### 4.6 추론 방법 및 시스템 적용

```
추론 시점: 30분마다 1회
출력: 재실 확률 p (0.0 ~ 1.0)

적용 규칙:
  p >= 0.8 이고 현재 EMPTY 상태
    → PRE_ARRIVAL 준비 (목표 온도 방향으로 서서히 이동)
  p < 0.8
    → 기존 상태 유지
```

### 4.7 콜드 스타트

시나리오 사전 학습 모델이 항상 존재 → 콜드 스타트 없음
실데이터 누적 후 자동으로 더 정확한 모델로 교체

## 5. 모듈 ③ — 퇴근 맥락 분류기

### 5.1 목적

현재 퇴근 감지는 고정 가중치 점수 합산이다.

```
현재: score = 인원감소(+30) + 외투(+25) + 기립(+10) + 시간대(+10)
      score >= 55 → PRE_DEPARTURE

문제: 가중치가 모든 환경·계절에 동일 적용
     → 여름엔 외투 안 입어도 퇴근함, 헬스장은 항상 서있음
```

학습된 분류기는 **환경과 패턴에 맞는 가중치를 스스로 찾는다**.

### 5.2 모델 구조

```
입력 (8개 features):
  people_count_delta  : 이전 대비 인원 변화량
  outerwear           : 외투 착용 여부 (0/1)
  activity_standing   : 기립 여부 (0/1)
  motion_score        : 최근 5분 평균 움직임 강도
  hour_sin, hour_cos  : 시간대 (sin/cos 인코딩)
  people_trend_5min   : 최근 5분 인원 추세 (slope)
  env_profile         : 환경 유형 (0~3, one-hot)

모델:
  Linear(8 → 64) → ReLU → Dropout(0.3)
  → Linear(64 → 32) → ReLU
  → Linear(32 → 1) → Sigmoid
  출력: 퇴근 확률 (0.0 ~ 1.0)
  파라미터 수: ~3,000개 (< 50KB)

손실 함수: Binary Cross Entropy
옵티마이저: Adam (lr=0.001)
```

### 5.3 학습 데이터 확보 전략 — 자동 레이블 생성

별도 레이블링 작업 불필요. **기존 상태 전이 로그에서 자동 생성**

```
로그에서 레이블 추출:
  STEADY → EMPTY 전이 발생
    → 전이 직전 30분 데이터 = 퇴근 신호 (label=1)

  STEADY → STEADY 유지 (퇴근 없음)
    → 동일 시간 데이터 = 비퇴근 (label=0)
```

**시나리오 데이터로 사전 학습 가능**:
```
퇴근 시나리오:
  17:30~18:30, 인원 감소 추세, 외투 착용률 증가 → label=1
  기타 시간, 인원 안정적              → label=0
```

### 5.4 학습 주기

```
초기: 시나리오 데이터로 사전 학습
운영 2주 후: 실제 퇴근 이벤트 100건 이상 누적 시 재학습
이후: 매주 일요일 새벽 3시 자동 재학습
```

### 5.5 추론 방법 및 시스템 적용

```
추론 시점: 매 VLM 분석 주기 (30초)
출력: departure_prob (0.0 ~ 1.0)

적용 규칙:
  departure_prob >= 0.7  → PRE_DEPARTURE 전환
  departure_prob <  0.3  → STEADY 복귀 (오탐 복귀)
  그 사이                → 현재 상태 유지 (hysteresis)
```

## 6. ML 운영 인프라 설계

### 6.1 ON/OFF 설정 구조

```python
# main.py 상단 — 테스트 환경 제어용
ML_PMV_LEARNING   = False  # ① PMV 선호도 온라인 학습
ML_OCCUPANCY_PRED = False  # ② 재실 패턴 예측 LSTM
ML_DEPARTURE_CLF  = False  # ③ 퇴근 맥락 분류기
```

```json
// models/ml_config.json — 런타임 설정 및 초기화
{
  "pmv_learning":          false,
  "occupancy_pred":        false,
  "departure_clf":         false,
  "reset_pmv_model":       false,
  "reset_occupancy_model": false,
  "reset_departure_model": false,
  "reset_all":             false
}
```

`reset=true` 항목은 실행 시 해당 `.pt`/`.pkl` 파일 삭제 후 자동으로 `false`로 복구

### 6.2 파일 구조

```
models/
  pmv_preference.pt        ← ① 학습된 가중치
  occupancy_lstm.pt        ← ② 학습된 가중치
  departure_clf.pt         ← ③ 학습된 가중치
  ml_config.json           ← ON/OFF + reset 플래그
  training_log.csv         ← 재학습 이력 (일시, loss, 교체 여부)

ml/
  pmv_model.py             ← ① 모델 정의 + 온라인 학습 로직
  occupancy_model.py       ← ② 모델 정의 + LSTM 추론 로직
  departure_model.py       ← ③ 모델 정의 + 분류 로직
  scenario_generator.py    ← 환경 프로파일 기반 시나리오 데이터 생성
  trainer.py               ← 새벽 재학습 스케줄러
```

### 6.3 새벽 재학습 스케줄러

```
매일 새벽 3시:
  1. hvac_system_performance.csv 에서 최근 30일 데이터 로드
  2. 데이터 충분 여부 확인
       ② LSTM:    30일치 이상 → 재학습 진행
       ③ 분류기:  퇴근 이벤트 100건 이상 → 재학습 진행
  3. 재학습 실행 (CPU 사용, GPU는 미사용)
  4. 검증 성능 비교
       개선됨 → 모델 파일 교체 + 로그 기록
       미개선 → 기존 모델 유지 + 로그 기록
  5. 완료 로그 저장
```

### 6.4 환경 전환 시 초기화 흐름

```
테스트 A 완료
  → ml_config.json: reset_all = true 설정
  → 다음 실행 시 모든 모델 파일 삭제
  → 환경 선택 화면 → 새 환경 선택
  → 새 환경의 시나리오 데이터로 사전 학습
  → 테스트 B 시작 (이전 학습 내용 완전 분리)
```

## 7. Sim-to-Real 전략 요약

실제 데이터 없이도 즉시 동작 가능한 이유:

```
배포 시:
  시나리오 데이터로 사전 학습된 모델 탑재
  → 설치 당일부터 ML 모드 동작 가능

운영 중:
  실제 데이터 누적 → 새벽 자동 재학습
  → 해당 공간·사용자에 특화된 모델로 점진적 개선

결과:
  초기:  시나리오 기반 범용 모델
  2주 후: 해당 환경에 맞게 최적화된 모델
  한 달 후: 사용자 개인 패턴까지 반영한 완전 특화 모델
```

이를 **Sim-to-Real Transfer + Continual Learning** 이라 부른다.

## 8. 기존 룰베이스와의 관계

ML 모듈은 룰베이스를 **즉시 교체하지 않는다**. 단계적으로 대체한다.

```
ML OFF (기본):
  퇴근 감지 → 기존 점수 합산 룰
  선호도   → 버튼 ±0.5 고정
  재실 예측 → 없음 (감지 후 반응)

ML ON (데이터 충분 후):
  퇴근 감지 → 학습된 분류기 출력
  선호도   → 학습된 MLP 예측값
  재실 예측 → LSTM 확률 기반 선제 대응

ML 실패 / 신뢰도 낮음:
  자동으로 룰베이스 fallback
```

## 10. 발표 핵심 메시지

> "VLM·YOLO는 사전학습 외부 모델을 활용하고,
> **열쾌적 개인화·재실 예측·퇴근 맥락 분류는 우리가 직접 설계·학습한 경량 모델**로 동작합니다.
>
> 실제 데이터가 없는 초기에는 환경 프로파일 기반 시나리오로 사전 학습하고(Sim-to-Real),
> 운영 중 수집된 실데이터로 매일 새벽 자동 재학습합니다(Continual Learning).
>
> 테스트 재현성을 위해 각 모듈을 독립 ON/OFF 및 초기화할 수 있으며,
> ML 실패 시 항상 기존 룰베이스로 자동 fallback됩니다."
